# End-to-end audit — goAML platform + Vyttah AML cockpit integration

**Date:** 2026-06-11
**Scope:** goAML repo (`feature/goaml-frontend-direct`, clean tree) + AML repo (`Backend_Java` and
`Frontend_Customer` on `feature/goaml-integration`).
**Method:** 8 parallel evidence-based auditors (intent-vs-reality, architecture+conventions, XSD/domain,
database, security, cross-repo contracts, testing+infra, frontends+go-live), each citing `path:line`,
read-only, static analysis (no suites executed this pass). Findings below were cross-checked where auditors
overlapped; converged findings are marked.

---

## Executive summary

**Overall health: strong core, leaky edges.** The standalone goAML platform is genuinely what the planning
record claims: all 14 phases + Phase 1.5 + the suite phases (A–D) verified built with code and tests, an
unusually clean architecture (engine/domain provably free of web/JPA imports, REST/MCP/CLI parity real,
zero TODO debt), and an XSD validation pipeline that genuinely guarantees **no unvalidated XML can reach
the FIU** — both `report_xml` writers validate first, and submit refuses non-VALID/APPROVED reports.

**The central systemic risk is the inverse guarantee: the gate ensures *validity*, not *completeness*.**
Because the lenient `t_person`/`t_entity` subject design makes most fields XSD-optional, several
independent defects each cause the FIU to receive a *clean-but-incomplete* report — data the user supplied
that silently never reaches the XML. This is a direct violation of the project's own "no lost field"
guardrail, and it's the theme uniting the worst findings.

**Top 5 risks:**

1. **Silent data loss into FIU filings** (Critical/High cluster, converged by 3 auditors):
   `countries.json` has 10 of 506 XSD country codes and the cockpit *silently omits* unresolvable
   nationalities/residences/addresses/identifications; `addressState` can never be populated so the
   subject address is omitted from **every** frontend-direct filing; failed relation-detail fetches degrade
   directors/UBOs to name-only or drop banks entirely with no warning; and the 7 mandatory
   LiveExShield-parity metadata fields are validated, then **discarded** on submit.
2. **The compliance gate is hollow on the live path** (High): every AML cockpit user JIT-provisions as
   goAML **MLRO** (`trusted_service.default_role=MLRO`, AML token endpoint not role-gated), tenant
   `review_required` defaults false, and `approve()` never checks reviewer ≠ author — one person can
   create→approve→submit to the FIU.
3. **No tenant-schema upgrade path** (Critical): tenant Flyway runs only at provisioning; any tenant
   provisioned before a later tenant migration breaks at runtime on the next release. Pre-production today,
   guaranteed incident the first time a real tenant exists across a release boundary.
4. **The browser builds regulator-bound XML payloads with zero test gate**: `Frontend_Customer` has no
   test runner at all (2,333 lines of DPMSR assembly), the wire keys are hand-maintained against
   Jackson's silently-ignore-unknown defaults, and AML Backend CI deploys with `-DskipTests`.
5. **Operational gaps before go-live**: committed fallback JWT secret boots fine if the env var is missed
   (token forgery → cross-tenant takeover); default 2-replica deployment double-polls the FIU and
   double-sends notifications (no lock); no backup/DR/data-retention policy anywhere (UAE AML
   record-keeping is a regulatory requirement); the pre-purge PII backup bundle still sits on disk.

**Go-live blockers** are mostly external by design (FIU/SACM registration, B2B creds, lookup exports, CSV
sign-off, AWS account) — but the items in the plan below under group (a) are *ours* and should land first.

**Biggest wins:** the XSD-first foundation (real FIU samples vendored and tested, golden regression,
full-field fidelity round-trip test), watertight tenant-isolation plumbing, exemplary Helm chart and
MLRO-gating UX in the goAML SPA, exact contract alignment on the federated auth seam, and a planning
record so auditable that every claimed SHA checked out.

## Dimension 3 — Domain & FIU/XSD fidelity

**Verified strong (validity):** `XsdSchemaValidator` compiles the real 5.0.2 XSD, XXE-hardened
(`engine/validation/XsdSchemaValidator.java:36-89`); `DpmsrReportBuilder.buildAndValidate` runs both
gates (`:61-67`); the only two writers of `report_xml` (`DefaultReportService.java:147-156`,
`GoamlXmlImporter.java:62-78`) both validate; submit refuses non-VALID/APPROVED
(`DefaultSubmissionService.java:63-67`). Activity shape, indicator mandatoriness
(`ReportValidator.java:188-193` against `goAMLSchema.xsd:312-318`), the `tax_reg_number` 1-char trap
(`goAMLSchema.xsd:1251-1258`), the mandatory phones wrapper, and `t_address` 5-field atomicity are all
consciously engineered around. Real FIU samples validate in `GoamlXsdValidationTest`;
`DpmsrFullFieldFidelityTest` round-trips an anonymized real report.

**Findings:**
- **[CRITICAL] Country lookup covers 10 of 506 XSD codes → silent field omission in filed XML.**
  `lookups/ae/countries.json` has AE/SA/GB/US/IN/PK/CN/DE/FR/CH only; XSD `country_type` has 506 enums
  (`goAMLSchema.xsd:3970`). The cockpit resolves every country against this lookup as authoritative and
  **silently omits unresolvable entries** (`CreateTransactionComponent.tsx:233-251` — "unresolvable
  entries are simply omitted") — a Filipino/Sri Lankan/Bangladeshi customer loses
  nationality/residence/address/identification from the filed XML, and the XSD gate cannot catch it
  (fields are optional on lenient types). `countries.json` is also missing from
  `LookupXsdConsistencyTest.java:30-53` (5 sets covered, not countries). Fix: regenerate countries.json
  from the XSD enumeration, add it to the consistency test, make the cockpit fail loudly on unresolved.
- **[HIGH] Curated path files `TPersonMyClient`, contradicting the lenient-subject rule.**
  `DpmsrRequestMapper.java:64,107-134` maps person parties to my_client — so every consumer of
  `POST /api/v1/reports/dpmsr` (MCP, accounting/screening integrations, CSV importer) hits the full
  my_client mandatory set. The CSV importer constructs persons with null gender/residence/taxRegNumber
  (`CsvImporter.java:124-128`) and can **never** produce a VALID person-party report. Caught by the gate
  (INVALID draft, no FIU exposure) but the feed is functionally broken, and `docs/15:58` enshrines the
  wrong choice. Fix: switch to lenient `GoamlParties.person(...)`; correct doc 15.
- **[HIGH] Mandatory metadata entered then destroyed** (converged: found independently by 3 auditors —
  see Dimension 7/10 for the cockpit side). The reconciliation doc promises "captured as report metadata"
  (`liveexshield-reconciliation.md:17-19`); as built it is entered-then-discarded.
- **[MEDIUM]** `personMyClient(...)` never maps `dto.address()` (`DpmsrRequestMapper.java:107-134` vs
  `DpmsrCreateRequest.java:74`) — a party address supplied on the curated path is silently dropped
  (XML stays valid → undetectable). One-line fix + mapper test.
- **[MEDIUM]** XSD name pattern `[a-zA-Z0-9 .'-]*` (`goAMLSchema.xsd:1043-1048`, `:858-865`) rejects
  `&`, `,`, `(`, `/`, and Arabic script — common in UAE legal names. Nothing pre-warns; the user gets a
  raw SAX message. Fix: business-rule normalizer + friendly error + UI input rule.
- **[MEDIUM]** my_client choice branches are annotated `goaml:supportedIn="5.2"`
  (`goAMLSchema.xsd:2730-2734`); whether the UAE FIU's deployed version accepts 5.2-only elements is
  UNVERIFIED — the curated path emits exactly these. Prefer baseline elements (the fix above resolves this).
- **[LOW]** `submit()` trusts persisted status+XML with no re-validation — cheap defense-in-depth: re-run
  the XSD validator on stored XML inside submit before zipping.

## Dimension 4 — Database design

**Verified strong:** isolation plumbing is textbook — signed `schema` claim → ThreadLocal → quoted
`SET search_path` with pool-release reset (`SchemaMultiTenantConnectionProvider.java:44,52,72-76`),
fail-closed default to `public`, context cleared/restored in `finally` at every entry point (filter,
poller, ingestion, the fixed `DefaultAuditService`). `tenant_goaml_person` one-ACTIVE is enforced by a
**partial unique index** (`shared/V4:32-33`); `external_identity` and `tenant_external_ref` uniqueness
exactly right; per-tenant `entity_reference UNIQUE` gives idempotency. No entity↔DDL drift in 5
spot-checks. AML's V102/V103 migrations are high quality (CHECK constraints, partial indexes, append-only
log).

**Findings:**
- **[CRITICAL] No upgrade path for existing tenant schemas.** Tenant Flyway is invoked **only** at
  provisioning (`DefaultTenantProvisioningService.java:98-105`, sole call site by repo-wide grep) despite
  `docs/07:15` and every migration header promising re-migration "on app upgrades". Tenant migrations
  V3–V7 arrived across Phases 8–D: any earlier-provisioned tenant lacks `attachment`, `notification`,
  `import_job`, `screened_subject`, `report.reviewed_by` → runtime `relation does not exist` the moment new
  code (poller included) touches it. Fix: startup `ApplicationRunner` iterating ACTIVE tenant schemas
  through the same programmatic Flyway before accepting traffic, + Helm readiness gating.
- **[HIGH] User hard-delete guard is not airtight.** `DefaultAdminService.java:144-148` checks only
  `report.created_by/reviewed_by` but the tenant schema also soft-references users from
  `attachment.uploaded_by` (`tenant/V3:18`), `import_job.created_by` (`tenant/V5:21`),
  `notification.recipient_user_id` (`tenant/V4:15`), `audit_log.actor_user_id` (`tenant/V1:14`); and the
  exists-check→delete is TOCTOU with no lock (no cross-schema FK to backstop). Fix: extend the guard,
  `SELECT … FOR UPDATE` on the user row, and/or default to disable with delete reserved for never-active users.
- **[MEDIUM]** Zero CHECK constraints in the whole goAML schema; status state machines
  (`report.status` incl. the undocumented `PENDING_REVIEW`/`APPROVED`, `submission.status`, …) are free
  text guarded only by app code. The AML repo does this right. Fix: CHECKs in the next migrations.
- **[MEDIUM]** `trusted_service.default_role` unconstrained text (`shared/V6:10`) — a column that can
  grant MLRO deserves `REFERENCES role(name)` or a CHECK.
- **[LOW]** Dead `refresh_token` table (created `shared/V2:91-99`, never written); throwaway
  `schema_baseline` ships to prod; `report.created_by/reviewed_by` and `submission.reportkey` unindexed;
  `submission` has no `submitted_by` (the acting MLRO lives only in audit_log — for compliance the attempt
  row should be self-contained); tenant deprovision lifecycle undocumented; `OffsetDateTime.now()` uses
  system zone (correct instant, cosmetic deviation).
- **AML side: [MEDIUM]** tenancy is row-level `company_id` with no RLS — isolation rests entirely on
  app-layer filtering (consistent with that codebase, but a weaker model than goAML's). **[LOW]**
  `goaml_transaction.customer_id` has no FK (polymorphic); `goaml_transaction_indicator` has no PK/unique
  (duplicate rows possible).

## Dimension 5 — Security

**Verified strong:** MLRO submit gating holds with parity at REST (`ReportController.java:157-158`), MCP
(`SubmissionTools.java:76-82`), CLI (`SubmitCommand.java:47`), re-guarded at the service layer. Every
controller endpoint is `@PreAuthorize`-gated or covered by `authenticated()`. JIT email-collision
account-takeover explicitly rejected (`DefaultFederatedAuthService.java:148-153`); disabled-user federated
block verified (`:119-121`); federated users get unusable random passwords; CORS deny-by-default; secrets
strategies clean; dev seeder hard-gated; audit coverage broad; no committed private keys in either repo;
goAML token held in-memory only in the cockpit.

**Findings (most severe first):**
- **[HIGH] Every authenticated AML user can become goAML MLRO and submit to the FIU.** The dev-seeded
  `trusted_service` has `default_role="MLRO"`, jit=true; `DefaultFederatedAuthService.java:155-156` honors
  it; AML's `GoamlTokenController.java:31-38` has no role gate (any logged-in cockpit user). The MLRO
  *check* applies but the *role* is granted wholesale — the gate is hollow on the live path. Whether a
  production trusted_service row would repeat this is UNVERIFIED, but V6 makes it one config row away.
  Fix: role-gate the AML token endpoint to compliance roles; default federated role ANALYST; MLRO only by
  explicit goAML admin action.
- **[HIGH] Committed fallback JWT secret** (`application.yml:98`, `dev-secret-please-replace…`) passes the
  32-byte check, so a deployment that forgets `GOAML_JWT_SECRET` boots and signs with a public-in-VCS key →
  forgeable SUPER_ADMIN/any-schema tokens = full cross-tenant takeover. Fix: refuse startup on the known
  default outside the dev profile.
- **[HIGH — push-gate] PII residue.** STATE.md:296-306 records the history purge as complete and verified
  (all SHAs rewritten, 0 PII-bearing objects). Two residues stand: (1) the pre-purge backup bundle
  **confirmed on disk** at `dev/goaml-prePII-backup.bundle`; (2) this audit did not independently re-scan
  history or open the (now-anonymized) samples — do one independent scan before the first remote push.
- **[MEDIUM]** Service assertions: `iat` optional so the 5-minute lifetime cap is skippable
  (`ServiceCredentialValidator.java:91-95` — a signer can mint years-long assertions), and **no replay
  protection** (no jti/nonce store; grep clean). Fix: require `iat`, enforce the cap unconditionally,
  jti seen-set (Redis already in the stack).
- **[MEDIUM]** Screening integration resolves the tenant from the **caller-supplied `companyId`
  parameter**, never cross-checked against the assertion's signed claims
  (`ScreeningIntegrationController.java:62-108`, `DefaultScreeningIngestionService.java:187-197`) — any
  valid SCREENING assertion can read filings/XML and create filings for *every* mapped tenant. Fix:
  require and match an `org` claim.
- **[MEDIUM]** AML `CurrentUserService.java:36-43` takes staff `companyId` from the client-controlled
  `X-Company-Id` header — combined with the MLRO finding, staff can JIT into any mapped goAML tenant.
- **[MEDIUM]** No user-status lookup or revocation on JWT auth (`JwtAuthFilter.java:55-62`): a
  disabled/deleted user keeps full access up to 15 minutes. Document or add a revocation cache.
- **[MEDIUM]** No AV/content scanning on attachments that enter the FIU submission ZIP
  (`DefaultAttachmentService.java:105-121`; deferred per `docs/09:109-110`); validation is
  filename/size/extension only. Promote to pre-go-live hardening.
- **[MEDIUM]** No rate limiting/lockout on `/api/v1/auth/login` or the federated exchange.
- **[LOW]** Tokens in `localStorage` in both SPAs (15-min TTL bounds it; no CSP either); `/actuator/prometheus`
  unauthenticated; `IllegalArgumentException` on malformed claims → 500 not 401; `iss` cross-check only
  if present; `/api/v1/auth/**` wildcard permitAll; AML `/internal/cron/sanctions/daily-run` is permitAll
  (unauthenticated state-changing trigger — flag to the AML team).

## Dimension 7 — Cross-repo integration (incl. the two-paths decision)

**Endpoint existence: all clear** — every goAML endpoint the AML repo calls exists; no phantom calls.
**Auth seam: aligned** — assertion claims (`aud="goaml"`, 120s TTL, iss/sub) match
`ServiceCredentialValidator` exactly; `FederatedTokenRequest`/`LoginResponse` mirrored exactly.
**Lookups: aligned by construction** — both paths read the same `LookupService` sets; item/status/
indicator/currency codes cannot drift. **Idempotency:** per-tenant unique `entity_reference` makes both
paths duplicate-safe individually; the same economic deal pushed via both paths creates two reports
(no shared key — inherent cost of keeping both).

**Contract findings:**
- **[HIGH] Path 2 metadata discard** (converged ×3). `CreateTransactionComponent.tsx:305-314` collects
  ~19 meta fields, `:881-888` blocks submit until 7 are filled, the UI claims "Captured with the
  transaction" (`:1176-1180`) — but `assemblePayload()` (`:856-874`) never includes meta, nothing persists
  in AML, and `setMeta({...EMPTY_META})` (`:916`) wipes it on success. *Entered-then-destroyed.*
  Fix: persist it — XSD-safe carriers already exist on the wire (`ReportPartyType.comments`,
  `TTransItem.comments`, report `reason`), or a metadata JSONB sidecar on the goAML report.
- **[HIGH] Path 1 `Goods.registrationDate` type mismatch = any dated deal unfileable.** AML sends
  `LocalDate` (`GoamlFilingPayload.java:43`) → goAML binds `OffsetDateTime`
  (`DpmsrCreateRequest.java:99`), no lenient Jackson config → 400 → rollback. Path 2 already solved this
  with `toOffset()` (`CreateTransactionComponent.tsx:292-294`). Fix: align the type on either side.
- **[HIGH] Path 1 silent field loss + fabricated fallbacks.** goAML's `ScreeningPartyMapper.java:66-95`
  drops person email/alias/identifications and entity trn/incorporationDate/address (the curated DTO can't
  carry them), and substitutes `countryOfBirth←nationality`, names→`"Unknown"`. The two paths file
  **materially different XML for the same customer**. Fix: route path 1 through the full-fidelity
  `DpmsrReportPayload`, replace substitutions with omission.
- **[MEDIUM]** Path 1 `FAILED` deal is terminal and pinned to a stale INVALID report (no
  `FAILED→DRAFT` transition; idempotency returns the old report forever —
  `DefaultScreeningIngestionService.java:114-118`). One INVALID filing permanently bricks a deal.
- **[MEDIUM]** Path 2's hand-maintained wire keys verified **all currently aligned** with the xjc types —
  but goAML's ObjectMapper silently ignores unknown keys, so one rename on either side silently drops a
  field from filed XML with zero error. Fix: a maximal round-trip contract test (POST → assert every field
  in `/detail` + XML) or `FAIL_ON_UNKNOWN_PROPERTIES` on this endpoint.
- **[MEDIUM]** AML session expiry mid-form is destructive: goAML token refresh is seamless
  (`axios.js:115-176`) but each refresh rides the AML instance whose 401 handler does
  `localStorage.clear()` + hard redirect (`axios.js:49-58`) — long KYC entry lost. Fix: re-auth modal +
  sessionStorage draft autosave.
- **[MEDIUM]** Review gate soft on path 2: JIT-MLRO (Dim 5) + `review_required` default false
  (`DefaultSubmissionService.java:63,185-187`) + `approve()` never checks actor ≠ author
  (`DefaultReportReviewService.java:59-67`).
- **[LOW]** Gender fallback emits first letter of unmapped values → non-XSD code (XSD allows `-`/`F`/`M`
  only); map unknown → `"-"`. Path 1 cannot send `location` (payload lacks the field goAML accepts).

**Two-paths recommendation: converge on path 2 (frontend-direct) as the only human filing UX; keep the
`/integration/screening/*` endpoints (not the AML deal module) for machine feeds.** Path 2 matches the
locked "goAML = single system-of-record" decision, carries full fidelity, and is the live-verified path;
path 1 has three contract hops and has already drifted (date bug, mapper losses, terminal-FAILED).
Pre-conditions before retiring path 1's UI: (1) fix the metadata discard so path 2 truly captures
everything; (2) restore approval rigor on path 2 (`review_required=true` for cockpit tenants + reviewer≠author
+ non-MLRO default federated role); (3) fix the machine-feed endpoint's date type and mapper losses for
whatever keeps calling it.

## Dimension 8 — Testing & coverage

**Verified strong (goAML):** ~91 backend test classes mirroring main packages; E2E flows prove the full
lifecycle (`ReportApiE2ETest.java:97`), RBAC denial, duplicate-ref conflict, full-fidelity persistence
(`:229`), federated auth incl. JIT + bad-signature 401, imports, admin, review queue; B2B stub-tested with
WireMock up to ZIP-submission E2E; the XSD chain (real samples → gate → builder → golden → E2E) is the
strongest possible static evidence; multi-tenant isolation test exists; SPA has exactly 77 specs incl.
dedicated `dpmsrPayload`/schema/form tests, Node 18 pinned in CI and README.

**Findings:**
- **[CRITICAL] Frontend_Customer has no test gate at all** — no runner, no test script, zero spec files —
  for 2,333 lines that build FIU-bound payloads in the browser (`CreateTransactionComponent.tsx` 1,253;
  `dpmsrPayload.ts` 255; `relationKyc.ts` 278). A silent mapping regression ships straight into
  regulator-bound XML. Fix: Vitest+RTL, port the goAML SPA's `dpmsrPayload.test.ts` golden-snapshot
  pattern, wire into CI.
- **[HIGH] AML Backend CI never runs tests** — only deploy workflows exist and both `mvn ... -DskipTests`
